Summary

"pyannote로 더 좋은 모델로 교체"는 우리에게 이미 절반 실현된 상태다. 코드 확인 결과 Minto2의 화자분리(FluidAudio offline)는 pyannote Community-1 모델(powerset segmentation + WeSpeaker 256d 임베딩 + VBx 클러스터링)을 CoreML로 변환한 것이다. senko조차 segmentation 모델 변환을 FluidAudio에서 가져다 썼다. 세 시스템은 경쟁이 아니라 공유 계보다.

따라서 진짜 병목은 "모델 품질"이 아니라 ① 화자 수 자동추정의 불안정성(Phase 2에서 측정·증명, 클러스터링 문제) ② provider가 FluidAudio에 강결합돼 교체 자유도가 없는 구조다. 단기 ROI가 높은 순서: min/max 화자 배선 살리기 → FluidAudio 버전 bump → provider 주입 디커플링. senko 채택과 pyannote Python 포팅은 우리 문제(온디바이스 회의록)에 손해다.

1. 세 시스템의 모델 계보

화자분리 파이프라인은 보통 4단계다: 분할(segmentation/VAD) → 특징추출 → 화자 임베딩 → 클러스터링. 세 시스템이 각 단계에 어떤 모델을 쓰는지 비교하면, 차이가 "모델 가족"이 아니라 "임베딩 모델과 클러스터링 알고리즘"에 국한됨이 드러난다.

쉽게 말하면

화자분리는 "녹음에서 누가 언제 말했는지"를 나누는 작업이다. ① 먼저 말이 있는 구간을 찾고(분할) ② 각 목소리를 256개 숫자로 된 지문으로 바꾼 뒤(임베딩) ③ 비슷한 지문끼리 묶어 "화자 1, 화자 2"로 그룹핑(클러스터링)한다. 세 제품은 ①의 모델을 사실상 같은 걸 쓰고, ②③에서만 갈린다.

2. 핵심 발견

발견 1 우리는 이미 pyannote 모델 위에 있다 — "모델 교체"는 절반 실현됨
.build/checkouts/FluidAudio/Sources/FluidAudio/ModelNames.swift:303

FluidAudio offline = "Pyannote Community-1 pipeline (powerset segmentation + WeSpeaker + VBx)"를 CoreML 변환한 것. HuggingFace repo FluidInference/speaker-diarization-coreml, variant "offline". 즉 사용자가 본 "pyannote의 좋은 모델"을 우리는 이미 쓰고 있다. 부족한 건 모델이 아니라 그 모델을 자유롭게 갈아끼우는 구조다(발견 3).

발견 2 진짜 병목은 클러스터링의 화자 수 추정 — 모델 품질이 아니다
Phase 2 측정: docs/work/2026-06-14-phase2-multispeaker-diarization-measurement.md

Fa/threshold 스윕은 비단조이고 녹음마다 최적 레버가 달라 단일 기본값으로 모든 회의를 못 맞춘다(4인→2, 6인→2, 위원회→4로 under-count). 반면 exactSpeakerCount 강제는 임의 N까지 정확. 이건 임베딩/분할 모델의 한계가 아니라 K(화자 수) 추정이라는 클러스터링 본질 문제다. senko의 HDBSCAN(밀도 기반, K 불필요)이 대안이지만, 사용자가 인원을 입력하는 우리 UX가 더 단순·견고하다.

발견 3 provider 추상화가 절반만 존재 — concrete 타입 강결합
Sources/Minto/Services/MeetingFileImportUseCase.swift:378 · Services/Diarization/SpeakerDiarizationProvider.swift:16

SpeakerDiarizationProvider 프로토콜은 있으나, 사용처가 let provider: FluidAudioOfflineDiarizationProvider로 concrete 타입을 직접 선언해 실질 교체 불가. pyannote식 "모델 교체성"을 우리 코드에서 얻으려면 모델이 아니라 이 주입 구조를 바꿔야 한다.

발견 4 min/max 화자 배선은 이미 있는데 UI가 안 쓴다 (dead 배선)
Services/Diarization/SpeakerDiarizationProvider.swift:30

생성자가 minSpeakers/maxSpeakers를 이미 노출하지만 UI는 exactSpeakerCount만 사용. "4~6명쯤"처럼 정확히 모르는 회의에 자유추정(불안정)도 exact 고정(틀리면 어긋남)도 아닌 중간을 줄 수 있는데 미활용.

발견 5 겹침 발화 정보를 powerset이 줄 수 있는데 matcher가 버린다
Sources/Minto/Services/TranscriptSpeakerMatcher.swift:10 (minimumOverlapRatio = 0.5)

powerset segmentation은 겹침을 검출 가능하나, 50% 겹침 규칙이 구간당 1명으로 뭉갠다(QA의 "2명인데 1명 표시" 현상의 원인). 회의록은 보통 발화당 1라벨로 충분해 체감 이득은 한계적 → 후순위.

3. 개선 방향 우선순위

한 줄 판단

사용자가 본 "pyannote 모델 교체성"의 진짜 가치는 모델을 바꾸는 것이 아니라 교체 가능한 구조다. 모델은 이미 pyannote 것을 쓰니, 단기엔 ②min/max 배선 + ③버전 bump이 ROI가 높고, "교체성" 자체가 목표면 ④provider 주입이 정공법이다(단 둘째 provider 계획이 설 때).

왜 "하지 말 것"인가

  • senko 채택: 강점이 NVIDIA 서버 속도(우리는 파일 배치라 비병목) + 겹침 미지원 + 깔끔한 num_speakers 부재. 우리가 얻을 게 없다.
  • pyannote Python 포팅: 온디바이스·Swift·ANE 이점을 통째로 잃는다. 회의 앱엔 손해.
  • Fa/threshold 추가 튜닝: Phase 2에서 비단조·녹음의존으로 막다른 길임이 증명됨.

4. 다음 작업 전 확인할 설계 제약

  1. provider 강결합: assignSpeakersIfNeeded()가 concrete 타입 직접 선언 → any SpeakerDiarizationProvider + 외부 주입 필요(방향 ④).
  2. diarizeWithSegmentsAndEmbeddings()가 internal: 모듈 외부·테스트 접근 불가 → 임베딩 추출 단위테스트 격리 불가(④에서 함께 해소).
  3. 실시간 경로 미지원: 녹음(TranscriptionViewModel)은 ChannelSpeakerLabeler(채널 기반)만 사용. 라이브 화자분리는 별도 파이프라인 설계 필요.
  4. 버전 표기 불일치: 메모리는 0.15.2, Package.swift는 from: "0.12.4" → Package.resolved로 실제 해소 버전 확인 후 ③ 진행.
