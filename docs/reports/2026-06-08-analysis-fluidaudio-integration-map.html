Summary

FluidAudio는 한 가지 STT 모델이 아니라 Swift/CoreML 기반의 오디오 도구 묶음이다. 우리 앱에는 전체를 한 번에 붙이기보다 Silero VAD, 오디오 타임라인, offline diarization 순서로 적용하는 것이 가장 안전하고 검증 가능하다.

한 줄 결론

추천 FluidAudio는 STT 교체보다 VAD와 diarization 개선 재료로 먼저 쓰는 게 맞다

현재 minto2는 이미 16kHz mono Float32 PCM을 만들고 있으므로 FluidAudio 입력 형식과 잘 맞는다. 다만 `TranscriptionViewModel`이 `VADProcessor`와 `STTService` concrete 타입에 직접 묶여 있고, `AudioChunk`/`Segment`에 실제 오디오 offset이 없어 speaker diarization을 바로 transcript 줄에 붙이기는 어렵다.

현재 구조

현재 요소 하는 일 FluidAudio 적용 시 문제
MicrophoneSource AVAudioEngine 입력을 16kHz mono Float32로 변환한다. 입력 형식은 맞지만 FluidAudio의 AudioConverter로 통일할지 결정해야 한다.
VADProcessor 에너지 dB 기준으로 speech/silence를 나누고 preview/final chunk를 만든다. Silero VAD를 넣으려면 같은 출력 계약을 가진 adapter가 필요하다.
STTService WhisperKit 모델을 다운로드/로드하고 final chunk를 전사한다. 현재는 WhisperKit 전용 class라 FluidAudio/SpeechAnalyzer/Nemotron을 같이 꽂기 어렵다.
Segment 전사 텍스트, 생성 시각, duration을 저장한다. speaker diarization과 맞추려면 실제 오디오 기준 start/end offset이 필요하다.

적용 후 목표 구조

기능별 설명

1순위 Silero VadManager: 말소리 구간을 더 똑똑하게 자르는 기능

무슨 기능인가: FluidAudio의 VadManager는 Silero VAD CoreML 모델로 256ms 단위 speech probability를 계산한다. 현재처럼 dB 에너지 하나만 보는 것이 아니라 모델이 학습한 말소리 패턴으로 speech/silence를 판단한다.

현재 어디에 붙나: VADProcessor 자리에 들어간다. 단, 기존 class를 바로 갈아끼우지 말고 VoiceActivityDetector protocol을 만들고 EnergyVADProcessor, SileroVADProcessor 두 구현을 병렬로 둔다.

기대 개선: 회의실 잡음, 조용한 발화, 짧은 대답에서 final chunk 생성 안정성이 좋아질 가능성이 있다. 개선 지표는 CER 하나가 아니라 short utterance recall, false positive, final empty count, stop-time flush 성공률이다.

주의: threshold를 낮추면 recall은 좋아져도 환각 전사가 늘 수 있다. 그래서 threshold 기본값을 바로 바꾸지 말고 같은 sample로 A/B해야 한다.

구조 참고 StreamingEouAsrManager: sliding window가 아니라 상태를 유지하는 streaming 구조

무슨 기능인가: 오디오 chunk를 순서대로 넣고, 내부 encoder/cache 상태를 유지하다가 EOU(end of utterance)나 finish()에서 final transcript를 만든다. 단순히 최근 8초/15초를 다시 전사하는 rolling window와 다르다.

현재 어디에 붙나: 당장 한국어 기본 엔진으로 쓰기보다 STTEngine protocol 설계에 반영한다. 현재 STTService.transcribe(pcmSamples:)는 stateless one-shot API라 streaming engine의 accept, finish, reset lifecycle을 표현하지 못한다.

기대 개선: 진짜 streaming ASR 엔진을 붙일 준비가 된다. 첫 partial latency, partial 수정 횟수, 마지막 chunk 누락 문제를 엔진 lifecycle에서 다룰 수 있다.

주의: Parakeet EOU 자체는 한국어 주력 후보가 아니다. 여기서 가져올 것은 모델보다 구조다.

제품 가치 OfflineDiarizerManager: 회의가 끝난 뒤 "누가 말했는가"를 붙이는 기능

무슨 기능인가: 전체 오디오를 segmentation, embedding, VBx clustering으로 처리해서 speaker timeline을 만든다. 회의록 앱에서는 CER만큼 중요한 speaker attribution 문제를 다룬다.

현재 어디에 붙나: 녹음 종료 후 저장/요약 전에 별도 background step으로 실행하는 것이 가장 안전하다. 실시간 UI에 먼저 넣지 말고, 저장된 transcript에 speaker label을 붙이는 PoC부터 시작한다.

기대 개선: "무슨 말이 있었는가"에서 "누가 그 말을 했는가"로 회의록 품질이 올라간다. 비싼 회의록 앱 대체라는 제품 목표에는 STT 모델 교체보다 체감 가치가 클 수 있다.

선행조건: 현재 Segment는 실제 오디오 offset이 없다. diarization timeline과 transcript를 겹쳐 보려면 AudioChunk.startTimeSeconds, Segment.audioStartSeconds, Segment.audioEndSeconds 같은 필드가 먼저 필요하다.

후순위 SortformerDiarizer: 실시간 speaker diarization 후보

무슨 기능인가: Sortformer는 streaming diarization 모델로, 고정 speaker slot에 대해 시간별 speaker probability를 낸다. 문서상 4 speaker slot, 낮은 latency preset, finalizeSession() 같은 session lifecycle을 제공한다.

현재 어디에 붙나: MicrophoneSource에서 나온 16kHz samples를 STT와 병렬로 diarizer에도 넣는다. 결과는 곧바로 transcript에 commit하지 말고 tentative speaker UI 또는 post-processing hint로 써야 한다.

기대 개선: 긴 회의에서 speaker 전환을 실시간으로 추적할 수 있다.

주의: 4명 초과, 겹침 발화, 조용한 원거리 발화에서는 잘못 붙을 수 있다. offline diarization을 먼저 검증한 뒤 실시간으로 확장하는 순서가 맞다.

기반 AudioConverter / ModelRegistry / offline-only: 입력과 모델 배포를 통제하는 기능

무슨 기능인가: FluidAudio는 모든 모델 입력을 16kHz mono Float32로 맞추라고 요구하고, 파일/버퍼 변환용 AudioConverter를 제공한다. 모델은 HuggingFace에서 자동 다운로드되지만 ModelRegistry.baseURL이나 offline-only 정책으로 앱 배포 방식을 통제할 수 있다.

현재 어디에 붙나: MicrophoneSource의 live buffer는 이미 16kHz Float32라 큰 변경이 적다. 파일 기반 benchmark, sample/meeting 처리, 모델 다운로드 UI는 FluidAudio 방식에 맞춰 AudioModelStore 계층으로 정리하는 것이 좋다.

기대 개선: 수동 WAV parsing 오류, 모델 캐시 꼬임, 앱 실행 중 예기치 않은 네트워크 다운로드를 줄인다. "완전 무료/로컬/프라이버시" 목표에는 offline-only가 특히 중요하다.

측정 필요 FluidAudio ASR 후보: 한국어 전사 엔진 교체는 아직 결론 내리면 안 된다

무슨 기능인가: FluidAudio에는 Parakeet, Qwen3-ASR, Cohere Transcribe, Streaming Nemotron 같은 ASR 계열이 있다. Qwen3-ASR과 Cohere Transcribe 문서에는 Korean 지원이 보이고, 최근 release note에는 Nemotron 3.5 multilingual 추가가 보인다.

현재 어디에 붙나: STTService를 직접 수정하지 말고 FluidAudioASREngine을 별도 구현으로 만든다. 기존 WhisperKit은 baseline으로 남겨야 한다.

기대 개선: 한국어 CER와 latency가 실제로 좋다면 WhisperKit보다 빠르거나 안정적인 후보가 될 수 있다.

주의: FluidAudio 문서 일부는 아직 Nemotron을 영어 중심으로 설명하고 release note는 multilingual을 말한다. 이 불일치 때문에 문서만 보고 기본값을 바꾸면 안 되고, sample/meeting 전체 CER/RTF/메모리로 검증해야 한다.

코드 변경 단위

변경 단위 무엇을 바꾸나 왜 필요한가 검증 기준
P1VAD protocol VoiceActivityDetector를 만들고 현재 VADProcessor를 EnergyVADProcessor adapter로 감싼다. Silero VAD와 기존 VAD를 같은 harness에서 A/B하기 위해 필요하다. 기존 VADProcessorTests 통과, sample VAD A/B 출력 생성
P1Silero adapter VadManager를 감싼 SileroVADProcessor를 추가한다. 4096 sample frame 누적과 flushPending() 계약을 맞춘다. 잡음/짧은 발화/stop 직전 누락 문제를 모델 기반 VAD로 비교하기 위해 필요하다. short utterance recall, false positive, final empty count, CER
P2오디오 타임라인 AudioChunk에 start/end seconds 또는 sample range를 추가하고, Segment에 audio offset을 보존한다. diarization timeline과 transcript segment를 겹쳐 speaker label을 붙이려면 필수다. segment timestamp monotonic, stop/drain 후 offset 보존, 저장 JSON 호환성
P3Offline diarization FluidOfflineDiarizationEngine을 추가하고 회의 종료 후 전체 audio 또는 disk-backed source로 speaker timeline을 만든다. 회의록에서 speaker label을 제공하기 위한 최소 리스크 경로다. speaker segment 수, overlap matching rate, 수동 speaker 품질 평가
P4STTEngine lifecycle transcribe one-shot 외에 accept, finish, resetSession, partial/final callback을 표현한다. 진짜 streaming engine은 내부 cache를 유지하므로 현재 stateless API로는 자연스럽게 표현되지 않는다. first partial latency, partial revision count, final CER
P5FluidAudio ASR bench Qwen3/Cohere/Nemotron 후보를 FluidAudioASREngine으로 감싸고 benchmark만 수행한다. 한국어 기본 엔진 교체는 실측 없이는 위험하다. sample/meeting 전체 CER, RTF, peak memory, crash-free run

적용했을 때 좋아지는 점

짧은 발화 누락 감소 가능성

Silero VAD가 energy threshold보다 speech probability를 더 잘 잡으면 "네", "맞아요", stop 직전 짧은 말이 chunk로 남을 가능성이 커진다.

잡음 환경 안정성 개선 가능성

회의실 팬 소리, 키보드, 배경 웅성거림을 단순 dB가 아니라 speech/non-speech 모델로 판단할 수 있다.

진짜 streaming 엔진 수용 가능

finish(), reset(), partial/final callback이 생기면 rolling window와 cache-aware streaming을 같은 UI 정책으로 다룰 수 있다.

speaker label 추가 가능

diarization을 붙이면 회의록이 단순 문장 목록이 아니라 발화자별 기록이 된다. 비싼 회의록 앱 대체 목표와 직접 연결된다.

모델 배포 정책 명확화

모델을 자동 다운로드할지, 앱 번들/로컬 캐시로 둘지, offline-only를 강제할지 제품 정책으로 분리할 수 있다.

ASR 후보 확장

WhisperKit만 보는 구조에서 벗어나 Apple Speech, FluidAudio, MLX sidecar를 같은 benchmark로 비교할 수 있다.

좋아지지 않는 것

오해 실제 판단
Silero VAD를 넣으면 CER가 무조건 좋아진다 아니다. VAD는 "어디를 전사할지"를 고른다. 너무 민감하면 잡음 chunk가 늘어 CER나 환각이 나빠질 수도 있다.
Diarization을 넣으면 STT 정확도가 좋아진다 아니다. diarization은 "누가 말했는지"를 붙인다. STT 텍스트 오류 자체를 고치지는 않는다.
StreamingEouAsrManager를 쓰면 바로 한국어 실시간 전사가 해결된다 아니다. 구조는 참고 가치가 크지만 한국어 모델/정확도는 별도 검증이 필요하다.
FluidAudio ASR이 있으면 WhisperKit은 버려도 된다 아니다. 현재 baseline과 같은 sample CER/RTF/메모리 비교를 통과해야 교체할 수 있다.

검증 계획

상태 실험 성공 기준
먼저 Energy VAD vs Silero VAD on sample/meeting 누락률이 줄고 false positive가 통제되며 final CER가 의미 있게 나빠지지 않는다.
다음 0.3s/0.5s/0.8s short utterance probe 0.5s 이상 짧은 한국어 대답 recall 개선, non-speech phantom 증가 없음.
선행 AudioChunk/Segment offset propagation 모든 final segment가 실제 audio start/end를 갖고 저장/요약/export에서 깨지지 않는다.
후속 Offline diarization on meeting sample speaker segment가 transcript segment와 overlap matching되고 수동 검토에서 speaker turn이 납득 가능하다.
보류 FluidAudio ASR 기본값 승격 현재는 미검증. sample/meeting 전체 CER, RTF, memory 결과가 WhisperKit/SpeechAnalyzer/Nemotron sidecar보다 좋아야 한다.

권장 작업 순서

  1. VoiceActivityDetector protocol을 만들고 기존 VADProcessor를 adapter로 감싼다.
  2. FluidAudio 의존성을 별도 branch에서 추가하고 SileroVADProcessor만 PoC한다.
  3. sample/meeting으로 VAD A/B 결과를 HTML 보고서로 남긴다.
  4. AudioChunk와 Segment에 실제 오디오 offset을 추가한다.
  5. OfflineDiarizerManager로 회의 종료 후 speaker timeline을 생성한다.
  6. speaker timeline과 transcript segment를 overlap으로 매칭해 저장/export 화면에 붙인다.
  7. 마지막으로 FluidAudio ASR 후보를 benchmark 전용 엔진으로 붙이고 CER/RTF/메모리만 비교한다.

근거: FluidAudio 문서와 최신성 주의
  • FluidAudio API 문서는 모든 module 입력이 16kHz mono Float32라고 설명한다.
  • API 문서는 VadManager가 Silero VAD CoreML로 4096 samples, 256ms frame을 처리한다고 설명한다.
  • API 문서는 StreamingEouAsrManager가 process, finish, reset, EOU callback lifecycle을 가진다고 설명한다.
  • API 문서는 OfflineDiarizerManager가 segmentation, embedding, VBx, timeline reconstruction pipeline을 수행한다고 설명한다.
  • Models 문서는 Qwen3-ASR과 Cohere Transcribe가 Korean을 포함한다고 설명한다.
  • Release note에는 v0.15.0에서 Nemotron 3.5 ASR Streaming Multilingual 0.6B가 추가됐다고 나오지만, Models 문서 일부는 아직 English-only Nemotron을 말한다. 따라서 Nemotron 계열은 문서만 믿지 말고 실제 package/API와 샘플로 확인해야 한다.
