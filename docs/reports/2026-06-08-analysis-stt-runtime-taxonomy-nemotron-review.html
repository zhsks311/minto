Summary

MLX, Core ML, whisper.cpp, WhisperKit은 같은 층위의 대체제가 아니다. Core ML과 MLX는 실행 런타임 계층이고, WhisperKit과 whisper.cpp는 Whisper 계열 모델을 제품에 붙이기 위한 라이브러리/SDK이며, Nemotron은 그 위에서 선택하는 ASR 모델 계열이다. 현재 실측 기준 Nemotron 8-bit는 정확도와 속도 모두 매력적이지만, 앱 기본값으로 승격하려면 Python sidecar 배포, Metal 권한, current VAD 5초/15초 chunk 재측정이라는 비용을 먼저 통과해야 한다.

1. 범주부터 다시 나누기

헷갈리는 이유는 이름들이 모두 “음성 인식” 근처에 있지만 실제로는 계층이 다르기 때문이다. 아래처럼 나누면 의사결정이 선명해진다.

이름 범주 무엇을 해주는가 우리 의사결정에서 의미
Core ML Apple 모델 실행 런타임 .mlmodel, .mlmodelc 모델을 CPU/GPU/Neural Engine에서 실행한다. Swift/macOS 앱 배포에 가장 자연스럽다. 모델 포맷 변환과 Core ML 컴파일 비용이 있다.
MLX Apple Silicon용 ML 배열/추론 프레임워크 unified memory, lazy evaluation, Metal 가속을 쓰는 NumPy 유사 프레임워크다. Nemotron처럼 MLX 커뮤니티 모델을 빠르게 실험하기 좋다. 앱 배포 계층은 직접 설계해야 한다.
WhisperKit Swift SDK + Whisper Core ML 실행기 Whisper 계열 Core ML 모델 다운로드, 로드, 전사 API를 제공한다. 앱 통합이 쉽고 fallback으로 좋다. MLX 모델을 직접 돌리는 도구는 아니다.
whisper.cpp C/C++ Whisper 런타임 Whisper 모델을 ggml/GGUF 계열 포맷으로 실행하고 Metal/Core ML encoder 가속도 지원한다. Python 없이 네이티브 통합하기 좋다. 단 Whisper 계열 안에서 움직인다.
Nemotron ASR ASR 모델 계열 이번 실험에서는 mlx-community/nemotron-3.5-asr-streaming-0.6b-8bit를 MLX Audio로 실행했다. 실측 정확도/속도는 좋다. WhisperKit에 넣는 모델이 아니라 MLX sidecar로 붙이는 편이 현실적이다.
MLX sidecar 앱 통합 아키텍처 Swift 앱 옆에 Python/MLX worker를 한 번 띄워 chunk 요청을 보내고 JSON 결과를 받는다. 가장 빠른 Nemotron 통합 경로다. 배포와 프로세스 관리가 추가된다.

2. 같은 말처럼 들리는 것들의 실제 차이

Core ML vs MLX

Core ML은 Apple 플랫폼 앱 배포를 위한 모델 실행 런타임에 가깝다. 모델을 컴파일된 번들로 만들고 시스템 런타임이 실행한다.

MLX는 Apple Silicon 연구/실험/추론을 위한 배열 프레임워크다. Python과 Swift API가 있지만, 특정 ASR 모델의 앱용 wrapper까지 자동 제공하는 것은 아니다.

WhisperKit vs whisper.cpp

WhisperKit은 Swift + Core ML 중심이다. iOS/macOS 앱에 자연스럽지만 Core ML Whisper 포맷을 전제로 한다.

whisper.cpp는 C/C++ 중심이다. Python 없이 붙이기 쉽고 Metal/Core ML 지원도 있지만, 기본적으로 Whisper 계열 모델 런타임이다.

MLX sidecar vs Swift MLX 직접 통합

MLX sidecar는 이미 동작하는 Python wrapper를 앱 밖 프로세스로 쓰는 방법이다. 빠르게 검증할 수 있다.

Swift MLX 직접 통합은 가능성은 있으나 Nemotron의 모델 로드, 전처리, generation wrapper를 Swift로 옮겨야 한다.

모델명 vs 런타임명

large-v3 turbo, Nemotron, Fun-ASR는 모델 또는 모델 계열이다.

Core ML, MLX, ggml, ONNX Runtime은 그 모델을 실행하는 런타임 또는 포맷 계층이다.

5. Nemotron 통합으로 얻는 장점

장점 1 한국어 회의 정확도 개선 가능성이 크다

전체 meeting corpus에서 Nemotron 8-bit weighted CER는 22.19%였다. 같은 집계의 WhisperKit turbo 45.91%, sherpa64 31.39%보다 낮다. 회의록 앱에서 가장 중요한 “나중에 고칠 말의 양”을 줄일 수 있다.

장점 2 Python sidecar여도 속도는 충분했다

10.2시간 오디오를 832.4초에 처리했다. Python이 병목이라기보다 MLX/Metal 추론이 주된 연산이다. worker를 매 chunk마다 새로 띄우지 않고 상주시킨다면 실시간 final 전사에는 속도 여유가 있다.

장점 3 Whisper 계열 밖 모델을 사용할 수 있다

WhisperKit과 whisper.cpp는 기본적으로 Whisper 계열에 강하다. Nemotron을 MLX sidecar로 붙이면 Whisper 구조에 묶이지 않고, 앞으로 한국어 회의에 더 좋은 MLX ASR 모델을 교체 실험할 수 있다.

장점 4 무료/로컬 우선 제품 방향과 맞다

사용자의 목표는 비싼 회의록 앱 대체, 완전 무료 또는 개인 OAuth 기반이다. Nemotron sidecar는 클라우드 STT 비용 없이 로컬 전사를 제공하므로 제품 철학과 맞다.

7. 권장 통합안

권장 기본 교체가 아니라 실험 엔진으로 붙인다

먼저 STTEngine protocol을 만들고 기존 WhisperKit을 WhisperKitSTTEngine으로 감싼다. 그 다음 NemotronSidecarSTTEngine을 추가한다. 초기 릴리스에서는 final chunk 전용으로만 사용하고, preview는 기존 WhisperKit 또는 비활성화 상태로 둔다.

단계 작업 성공 기준
1 STTEngine 경계 추가 기존 WhisperKit 동작이 바뀌지 않고 기존 테스트 통과
2 Nemotron worker 상주 프로세스 구현 모델 1회 로드, chunk 요청/응답 id 매칭, timeout 처리
3 current VAD chunk 기준 재측정 5초/15초 chunk에서 CER와 latency가 WhisperKit보다 유의미하게 우수
4 stress test 1시간 이상 녹음, worker restart, 메모리 peak, fallback 경로 검증

8. 최종 판단

판단 Nemotron은 붙일 가치가 있지만, 앱 기본값 승격은 아직 이르다

정확도와 처리 속도는 충분히 매력적이다. 다만 현재 결과는 offline 60초 chunk이고, 통합 방식은 Python/MLX sidecar다. 제품 기본값으로 바꾸기 전에는 current VAD chunk benchmark와 장시간 안정성 검증이 필요하다.

추천 결론 WhisperKit은 안정 fallback, Nemotron은 고정확도 실험 엔진

WhisperKit은 Swift/Core ML 네이티브 통합 비용이 낮다는 장점이 있다. Nemotron은 정확도와 모델 확장성이 장점이다. 둘을 한쪽으로 단정하지 말고 STTEngine 경계 아래에서 선택 가능하게 만드는 것이 가장 검증 가능한 접근이다.
