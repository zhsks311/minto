Summary

이번 실험의 결론은 간단하다. 지원되는 macOS에서는 Apple SpeechAnalyzer를 먼저 붙이고, WhisperKit은 fallback으로 남기는 방향이 가장 타당하다. BatiSay는 공개 모델 파일이 없어 아직 테스트할 수 없고, Nemotron MLX는 숫자는 좋지만 앱 통합 비용이 더 크다.

한눈에 보는 결론

낮은 CER가 좋다

CER는 글자 오류율이다. 12.3%는 25.1%보다 같은 자막 기준에서 덜 틀렸다는 뜻이다.

낮은 RTF가 빠르다

RTF 0.006은 120초 오디오를 약 0.77초에 처리했다는 뜻이다. 1보다 낮으면 실시간보다 빠르다.

통합 난이도도 중요하다

숫자가 좋아도 앱에 안정적으로 넣기 어려우면 바로 기본값으로 바꾸기 어렵다.

주요 발견

Finding 1 SpeechAnalyzer가 현재 가장 좋은 다음 후보
docs/stt-parallel-poc-result.md

현재 환경에서는 ko-KR가 ko_KR로 잡히고, 120초 clipped WAV에서 CER 12.3%를 냈다. 같은 기준의 WhisperKit 25.1%보다 훨씬 낮다.

Finding 2 SpeechAnalyzer preview는 그대로 쓰면 흔들린다
/private/tmp/minto2_speech_analyzer_stream_bench.swift

streaming input은 동작했지만 volatile 결과가 글자 단위로 많이 바뀌었다. live preview에 넣으려면 debounce, range 조립, final 확정 규칙이 필요하다.

Finding 3 Nemotron은 숫자는 좋지만 앱 통합이 무겁다
mlx-community/nemotron-3.5-asr-streaming-0.6b

8-bit 모델은 120초 CER 16.4%, RTF 0.030까지 나왔다. 하지만 WhisperKit 모델이 아니고 mlx-audio GitHub main이 필요해서 당장 Swift 앱 기본 엔진으로 넣기에는 부담이 있다.

Finding 4 BatiSay는 지금은 테스트 대상이 아니다
batiai/batisay-ko-base

AutoModel.from_pretrained(...)와 pipeline 모두 config discovery 단계에서 실패했다. 공개 repo에 config.json, model.safetensors, ggml, ct2 파일이 없어서 실제 추론이 불가능하다.

다음 작업 순서

  1. STTEngine seam을 먼저 추가한다. 기존 WhisperKit 동작은 그대로 두고, ViewModel이 concrete STTService에 직접 묶이지 않게 한다.
  2. SpeechAnalyzerSTTEngine을 별도 adapter로 붙인다. macOS availability gate를 두고, 불가능하면 WhisperKit fallback으로 간다.
  3. preview 조립 정책을 따로 만든다. volatile 결과를 그대로 UI에 뿌리지 말고 debounce와 final 확정 규칙을 둔다.
  4. Nemotron은 sidecar research lane으로 유지한다. 숫자는 좋지만 배포, 라이선스, Python 의존성, API 안정성을 더 봐야 한다.
  5. BatiSay는 모델 파일이 생길 때까지 멈춘다. 지금은 더 테스트해도 CER/RTF를 만들 수 없다.
