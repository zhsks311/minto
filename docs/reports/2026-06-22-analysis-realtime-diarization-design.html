실시간 화자분리 설계 — "앵커드 하이브리드"

회의 중 "누가 말하는지"를 실시간으로 보여주되, M3 Pro 맥북에서 돌아가고 최대한 정확하게.

회의 중에는 빠른 AI(LS-EEND)로 "지금 누가 말하나"를 실시간 표시하고, 회의가 끝나 저장할 때는 더 정확한 AI(VBx)로 다시 한 번 정리해 라벨을 확정한다. 그리고 "내 목소리(마이크)"와 "전에 등록해 둔 사람들(보이스프린트)"을 닻(anchor)으로 미리 알고 시작해서, 맨바닥에서 추측하는 것보다 훨씬 정확하게 만든다. 받아쓰기는 ANE 칩, 화자분리는 CPU로 나눠 돌려 M3 Pro에서도 끊김 없이 돌아간다.

1. 그래서 무슨 문제를 푸는 거예요?

화자분리(diarization)는 회의 녹음에서 "누가 언제 말했는지"를 알아내는 일입니다. 받아쓰기(STT)가 "무슨 말을 했나"를 풀면, 화자분리는 "누가 했나"를 풀어요. 둘을 합치면 "철수: 안녕하세요 / 영희: 네 반가워요" 같은 회의록이 됩니다.

12살도 이해하는 비유

녹음을 들으면서 목소리마다 색깔 스티커를 붙인다고 생각해 보세요. 철수 말엔 파란 스티커, 영희 말엔 빨간 스티커. 그런데 (1) 처음엔 누가 누군지 모르고, (2) 비슷한 목소리는 헷갈리고, (3) 회의가 끝나기 전에 실시간으로 붙여야 하면 더 어렵습니다. 우리가 풀려는 게 바로 이 "실시간으로, 정확하게 스티커 붙이기"입니다.

왜 "실시간"이 특히 어렵나요?

연구가 분명히 말합니다: "실시간(스트리밍) 화자분리와 나중에 처리(오프라인)의 정확도 차이는, 받아쓰기에서의 차이보다 훨씬 크다." 즉 실시간은 미래(앞으로 무슨 말이 나올지)를 못 보고 지금까지만 보고 정해야 해서 본질적으로 부정확합니다. 그래서 "실시간만" 고집하면 정확도를 크게 잃습니다.

비유

생중계 vs 녹화 편집과 같아요. 생중계(실시간)는 즉시 보여주지만 실수가 있고, 녹화 후 편집(오프라인)은 처음부터 끝까지 다 보고 고칠 수 있어 깔끔합니다. 우리는 둘 다 하기로 했어요 — 생중계로 일단 보여주고, 끝나면 편집본으로 바꿔치기.

2. 고른 방법 — "앵커드 하이브리드"

이름이 어렵지만 세 조각입니다: ① 실시간 패스(빠른 AI) + ② 최종 패스(정확한 AI, 저장 때) + ③ 앵커(아는 사람 미리 알기). 그림으로 보면:

위: 녹음 중 빠른 실시간 처리(임시 라벨). 아래: 저장할 때 정확한 VBx로 다시 정리해 확정. "나"와 등록 화자는 ⚓앵커로 양쪽에 도움.

① 실시간 패스 — 빠른 AI (LS-EEND, CPU에서)

녹음하는 동안 LS-EEND라는 가벼운 AI가 100밀리초(0.1초)마다 "지금 누가 말하나"를 추측해 화면에 임시 라벨을 띄웁니다. 최대 10명까지 구분할 수 있어 사람이 많은 회의도 커버합니다.

③ 앵커 — "아는 사람"을 미리 알고 시작하기 ⚓ (이게 핵심!)

왜 이게 비밀 무기인가

보통 화자분리는 아무것도 모르는 상태에서 "이 목소리들 중 몇 명이고 누가 누구지?"를 맨바닥부터 추측합니다. 그런데 우리는 이미 아는 사람이 있어요: (1) 내 마이크 소리는 100% "나"입니다(따로 들어오니까), (2) 전에 등록해 둔 사람들(보이스프린트)의 목소리 지문이 있습니다. 이 둘을 닻(anchor)으로 박아두면, AI는 "나머지 모르는 사람만" 추측하면 되니 훨씬 정확합니다. 마치 퍼즐의 모서리 조각 몇 개를 미리 맞춰놓고 시작하는 것과 같아요. 이걸 학계에선 Target-Speaker VAD라 부르고 2024년 최신 정확도 향상 기법입니다.

② 최종 패스 — 정확한 AI (VBx, 저장할 때)

회의가 끝나 저장하는 순간, 저장된 오디오 전체를 더 정확한 VBx로 처음부터 끝까지 다시 분석합니다(미래까지 다 볼 수 있으니 더 정확). 그리고 앵커(아는 사람)와 사용자가 입력한 예상 인원으로 한 번 더 도와줍니다. 그 결과로 실시간에 띄웠던 임시 라벨을 확정 라벨로 바꿔치기합니다 — 이걸 재조정(reconciliation)이라 해요.

왜 굳이 두 번 하나요?

연구가 확인해 줬습니다 — 정확한 VBx는 실시간으로는 못 씁니다("VBx를 온라인에 쓰면 성능이 크게 떨어진다"). 그래서 실시간엔 빠른 LS-EEND, 저장 땐 정확한 VBx로 역할을 나누는 게 정답입니다. 실제 시스템들도 "온라인으로 대충 → 끝나면 다시 정리(resegmentation)"라는 2단계로 오프라인급 정확도에 근접합니다.

3. M3 Pro 맥북에서 정말 돌아가요?

네 — 단, 일을 나눠 줘야 합니다. 맥북 칩에는 일 잘하는 부서가 여러 개 있어요: ANE(AI 전용), CPU(범용), GPU. 핵심은 받아쓰기와 화자분리가 같은 부서(ANE)를 동시에 차지하지 않게 하는 것입니다.

받아쓰기는 ANE, 실시간 화자분리는 CPU로 분리 → 녹음 중 경합 없음. 무거운 VBx는 녹음이 끝난 뒤에만 돌아 라이브에 부담 0.

근거: FluidAudio는 애플 칩에서 1시간 오디오를 수십 초에 처리(M1에서 60배속)하고, LS-EEND는 "M4 MAX CPU만으로도 ANE급 속도"라고 문서가 밝힙니다. M3 Pro는 그 사이라 충분히 가능합니다. 무거운 VBx는 녹음 중이 아니라 저장 시점에만 돌아서 실시간 부담이 아예 없습니다.

솔직한 단서

"가능하다"는 문서·벤치마크 기반 추정입니다. M3 Pro에서 받아쓰기+화자분리를 동시에 돌렸을 때 실제 속도(RTFx)와 발열·배터리는 직접 측정해 확인해야 합니다(다음 단계). 또 LS-EEND를 CPU로 돌릴 때 정확도/속도가 ANE와 같은지도 검증 대상입니다.

4. 왜 이게 "최대한 정확"한가요? — 3가지 정확도 장치

장치쉽게근거
2번 처리(하이브리드)생중계로 보여주고, 끝나면 편집본으로 교체오프라인이 스트리밍보다 정확(VBx DER 12% vs LS-EEND 25.7%)
앵커(아는 사람 고정)퍼즐 모서리 먼저 맞추기 — "나"+등록 화자를 닻으로Target-Speaker VAD = 2024 SOTA 정확도 기법
예상 인원 입력"오늘 4명이야"라고 알려주면 AI가 안 헤맴우리 측정: exactSpeakerCount가 가장 견고한 레버

5. 핵심 용어 사전 📖

화자분리 diarization

녹음에서 "누가 언제 말했는지" 목소리마다 이름표를 붙이는 일. (받아쓰기 = "무슨 말", 화자분리 = "누가")

받아쓰기 STT

Speech-to-Text. 말소리를 글자로 바꾸는 것. 우리는 WhisperKit을 씀.

스트리밍 / 온라인 streaming

녹음하는 중에 실시간으로 처리. 빠르지만 미래를 못 봐서 덜 정확. (생중계)

오프라인 / 배치 offline

녹음이 다 끝난 뒤 전체를 한 번에 처리. 느리지만 정확. (녹화 편집)

하이브리드 2-pass hybrid

실시간(빠름) + 저장 시(정확)를 둘 다 쓰는 방식. 우리가 고른 길.

LS-EEND

가벼운 실시간 화자분리 AI. 최대 10명, 0.1초마다 갱신. CPU로도 돌아감. 우리 라이브 엔진.

Sortformer

또 다른 실시간 AI. 4명까지지만 "누가 누구인지"를 안 흔들리게 잘 유지. (지금은 인원 제한 때문에 보류, 라이브 등록용 후보)

VBx

가장 정확한 오프라인 화자분리(묶기) 방법. 실시간엔 못 쓰고, 저장 시 최종 정리에 사용.

앵커 / 타깃 화자 anchor / TS-VAD

이미 아는 사람의 목소리를 미리 알고 시작해 정확도를 높이는 기법. 우리는 "내 마이크"와 "등록 화자"를 닻으로 씀.

보이스프린트 voiceprint

한 사람의 목소리를 256개 숫자로 만든 "목소리 지문". 등록해두면 다음 회의에서 그 사람을 알아봄. (우리가 만든 기능)

재조정 reconciliation

실시간에 붙인 임시 라벨을, 저장 시 정확히 계산한 최종 라벨로 바꿔치기하는 과정.

채널 channel

소리가 들어오는 통로. 우리는 마이크(=나)와 시스템 소리(=상대)를 따로 받음. 그래서 "나"는 항상 알 수 있음.

ANE

Apple Neural Engine. 맥북 칩 안의 AI 전용 부서. 받아쓰기를 여기서 돌림.

DER

Diarization Error Rate. 화자분리가 얼마나 틀렸는지(%). 낮을수록 좋음. VBx 12% vs LS-EEND 25.7%.

RTFx

실시간 대비 처리 속도. 60x면 1시간 오디오를 1분에 처리. 1보다 크면 실시간 가능.

화자 수 / exactSpeakerCount

회의 인원. 사용자가 "4명"이라고 알려주면 AI가 안 헤맴 — 우리가 찾은 가장 강력한 정확도 레버.

6. 다음 단계 & 따로 적어둔 것

  • ADR 0005 작성·검토 완료 ✅ — 전문가 2명(아키텍처·비평) 검토 통과. 단 아래 "M3 Pro 테스트"가 통과해야 정식 수락·구현 시작.
  • M3 Pro 실측(다음 게이트): 받아쓰기+LS-EEND 동시 실행 시 속도(RTFx>1배)·발열·배터리, CPU 정확도. 이걸 통과해야 만들기 시작.
  • 재조정 UX 설계: 임시→확정 라벨이 바뀔 때 사용자가 혼란 없게 — 구현 전 필수.

7. 전문가 검토로 더 똑똑해진 점 (쉽게)

검토가 잡아준 핵심

설계를 AI 검토자 2명(아키텍처 보는 눈 / 깐깐하게 트집 잡는 눈)에게 맡겼더니, 혼자선 못 본 것들을 잡아줬어요. 가장 중요한 4가지를 쉽게 풀면:

  1. "앵커는 사실 두 가지 다른 일을 한다." ① 화면 라벨이 덜 깜빡이게 하는 일(실시간 편함), ② 최종 정확도를 높이는 일. 그런데 저장되는 녹음 파일은 모든 목소리가 한 트랙으로 섞여서 저장돼요 — "이건 내 마이크였어"라는 구분이 섞이면서 사라집니다. 그래서 최종 정리(VBx)에 도움을 주려면, "이 구간은 나였다"는 정보를 따로 옆에서 알려줘야 합니다. 그냥 파일만 넘기면 앵커가 최종엔 안 닿아요.
  2. "두 AI가 같은 일꾼을 동시에 붙잡지 않게." 받아쓰기와 화자분리가 컴퓨터의 같은 부서를 두고 싸우면 끊깁니다. 그래서 화자분리는 별도 작업 공간(actor)에서 혼자 돌리고, 결과(라벨)만 화면에 전달하도록 분리합니다.
  3. "무리면 더 단순한 길로 후퇴할 수 있게 미리 만든다." 혹시 M3 Pro가 실시간 신경망을 버거워하면, 신경망을 빼고 채널(나/상대)만으로 라이브를 보여주고 저장 시 정확히 정리하는 더 가벼운 길로 갈아탈 수 있게 설계를 미리 갈아끼움 가능하게 만듭니다. (정확도의 핵심인 "저장 시 VBx"는 어느 길이든 유지)
  4. "라이브 라벨을 안전하게 바꾼다." 라이브에서 본 "철수"가 저장 후 바뀌면 "어? 틀렸네" 싶을 수 있어요. 그래서 사용자가 직접 고친 라벨은 절대 안 건드리고, 바뀌는 부분은 자연스럽게 보여주는 화면을 먼저 설계합니다.
솔직한 현재 상태

설계와 검토는 끝났지만 아직 "만들기 시작" 전입니다. M3 Pro에서 실제로 받아쓰기+화자분리를 동시에 돌려보는 테스트(스파이크)를 통과해야 구현에 들어갑니다 — "될 것 같다"가 아니라 "직접 재서 된다"를 확인하는 게 정직한 다음 단계예요.

나중에 볼 다른 트랙 (사용자 요청으로 기록)

트랙 B — 오프라인 정확도 더 끌어올리기: 이미 VBx로 충분히 좋아(DER 12%) 한계효용 작음. 임베딩 교체(ERes2NetV2 등)는 측정상 비병목이라 보류.

트랙 C — 회의 간 화자 기억(보이스프린트 확장): 등록 화자를 여러 회의에 걸쳐 자동 인식. 위 "앵커"와 직접 연결됨 — 앵커드 하이브리드를 만들면 이 트랙도 같이 강해짐. 크로스세션 정확도 측정이 선행 과제.

근거 / 출처 (연구 검색)

  • LS-EEND (arXiv 2410.06670) — 온라인 attractor, 유연한 화자 수
  • Target-Speaker VAD: MIMO-TSVAD (arXiv 2401.08052), PET-TSVAD (Microsoft ICASSP 2024), Online Neural Diarization w/ Target Speaker Tracking (IEEE 2024)
  • 2-pass 재조정: online clustering + Bayesian resegmentation (Springer 2023) — 오프라인급 정확도+저지연. "VBx는 온라인 부적합" 확증
  • 온디바이스: FluidAudio(애플 ANE), speech-swift(MLX+CoreML) — M3급 동시 ASR+diarization 실현 사례
