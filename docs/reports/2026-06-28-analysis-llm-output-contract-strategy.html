업무 전략이 아니라 출력 계약
Summary

기존 구조는 여러 LLM을 끼우기 좋은 provider adapter 계약을 이미 갖고 있다. 부족한 부분은 provider 선택 전략이 아니라, “이 요청은 구조화 JSON이어야 한다”는 출력 계약을 공통 타입으로 표현하지 못한다는 점이다.

한 줄 결론

권장 provider 업무 전략은 늘리지 말고 출력 형식 adapter만 추가한다

provider가 “회의 요약”, “교정”, “검색 답변” 같은 업무 맥락을 알기 시작하면 결합도가 커진다. 반대로 provider가 plainText와 jsonSchema 같은 출력 형식만 자기 API 문법으로 바꾸면 기존 확장 구조를 유지한다.

현재 구조

현재 계약이 잘하는 일

  • 새 provider를 LLMTextGenerationProvider로 붙인다.
  • 호출자는 generateText 하나만 알면 된다.
  • LLMProviderRegistry가 provider 생성을 숨긴다.

현재 계약이 놓치는 일

  • LLMTextRequest에 출력 형식 필드가 없다.
  • finalSummary의 JSON 요구가 prompt와 provider 구현에 흩어진다.
  • provider별 structured output 문법 차이가 테스트 표면으로 올라오지 않는다.

도입 범위

넣을 것

  • LLMOutputFormat: plainText, jsonSchema
  • LLMTextRequest.outputFormat: 호출자가 원하는 출력 계약
  • MeetingSummarySchema: 한 곳에서 공유하는 schema builder
  • provider adapter별 output format 변환 테스트

넣지 않을 것

  • FinalSummaryStrategy 같은 업무별 provider 전략
  • provider registry 전면 개편
  • 교정/검색 답변 로직의 동시 재작성
  • 계정 로그인 provider의 무리한 schema 강제

왜 업무 전략이 아닌 출력 계약인가

구분 업무 전략을 provider에 넣는 방식 출력 계약을 request에 넣는 방식
provider가 아는 것 회의 요약, 교정, 검색 답변의 업무 의미 plain text인지 JSON schema인지
결합도 앱 기능이 늘 때 provider가 같이 변한다 새 출력 형식이 생길 때만 provider가 변한다
테스트 포인트 provider x useCase 조합이 커진다 provider x outputFormat 조합만 본다
이번 문제와의 관계 문제를 가릴 뿐 누락을 막기 어렵다 schema 누락을 request 계약에서 바로 잡는다

작은 설계 스케치

public enum LLMOutputFormat: Sendable, Equatable {
    case plainText
    case jsonSchema(LLMJSONSchema)
}

public struct LLMJSONSchema: Sendable, Equatable {
    public let name: String
    public let schema: [String: JSONValue]
    public let strict: Bool
}

public struct LLMTextRequest: Sendable {
    public let useCase: LLMUseCase
    public let instructions: String
    public let userContent: String
    public let outputFormat: LLMOutputFormat
}

권장 단계

1 schema builder를 한 곳으로 이동

Ollama와 Gemini에 복사된 MeetingSummary schema를 공유 타입으로 옮긴다. 이 단계는 동작을 바꾸지 않고 중복만 줄인다.

2 LLMTextRequest에 outputFormat 추가

SummaryService는 최종 요약에서 .jsonSchema(MeetingSummarySchema)를 요청한다. provider는 업무 맥락 대신 출력 형식만 본다.

3 provider별 변환 테스트 추가

OpenAI, Gemini, Ollama, OpenRouter, Claude의 request body 테스트를 각각 둔다. 계정 로그인과 CLI는 지원 여부를 명시하고 prompt-only fallback을 문서화한다.
