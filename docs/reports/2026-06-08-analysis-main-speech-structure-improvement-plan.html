Summary

main의 새 구조는 모델 선택 UI에서 엔진 선택 UI로 한 단계 올라갔기 때문에, 개선 작업을 시작하기 좋은 상태다. 다만 아직 STTService가 모든 엔진 구현을 직접 품고 있고, VAD와 오디오 타임라인 seam이 없어서 Silero VAD, FluidAudio ASR, diarization은 바로 꽂기보다 단계별 구조 보강이 필요하다.

현재 main에서 달라진 점

영역 main 상태 판단
엔진 선택 모델 SpeechEngineID, SpeechEngineFamily, SpeechEngineAvailability가 추가됨 좋음 UI/설정 기준은 생겼다.
실제 엔진 실행 STTService.transcribe가 WhisperKit, SpeechAnalyzer, SFSpeech를 switch로 직접 분기 보강 필요 새 엔진을 계속 추가하면 class가 비대해진다.
Preview 정책 Apple 엔진은 supportsPreviewTranscription == false로 preview를 끔 안전 다만 final-only UX 표시가 필요하다.
VAD 기존 energy 기반 VADProcessor 유지 분리 필요 Silero VAD A/B를 하려면 protocol이 필요하다.
오디오 타임라인 AudioChunk와 Segment에 실제 audio offset 없음 선행 필요 diarization과 정확한 timestamp의 전제다.
Benchmark 수동 benchmark가 STT_MODEL 기반 WhisperKit 중심 바로 개선 가능 STT_ENGINE 기반으로 확장해야 한다.

적용 가능성 매트릭스

상태 개선 항목 왜 가능한가 / 왜 보류인가 적용 시 개선
바로 가능 Benchmark를 STT_ENGINE 기준으로 확장 main에 loadSpeechEngine과 SpeechEngineID가 이미 있다. WhisperKit, SpeechAnalyzer, SFSpeech를 같은 sample에서 비교 가능
바로 가능 엔진 availability 테스트 seam 추가 현재 availability는 OS/권한/asset 상태에 직접 의존한다. Settings disabled row, 권한 필요, fallback 정책을 deterministic test로 검증
바로 가능 Apple 엔진 final-only UX 문구 보강 Apple 엔진은 preview를 꺼둔 상태라 사용자가 "왜 실시간 문구가 없지"라고 느낄 수 있다. 사용자가 엔진별 preview/final 차이를 오해하지 않음
바로 가능 STTService 내부 엔진 구현 분리 동작은 유지하면서 WhisperKitSpeechEngine, AppleSpeechAnalyzerEngine, SFSpeechOnDeviceEngine로 이동 가능하다. FluidAudio, MLX sidecar, future engine을 추가해도 main service가 커지지 않음
구조 보강 후 가능 Silero VAD 도입 VADProcessor concrete가 ViewModel에 직접 박혀 있다. 짧은 발화 누락, 잡음 false positive, final empty count를 측정 기반으로 개선 가능
구조 보강 후 가능 Offline diarization 현재 transcript segment에 실제 audio offset이 없고, 전체 raw audio 저장 경로가 없다. 회의록에 speaker label을 붙일 수 있음
구조 보강 후 가능 true streaming ASR lifecycle 현재 API는 one-shot transcribe(pcmSamples:) 중심이다. rolling preview가 아니라 cache 유지형 streaming 엔진을 수용 가능
측정 전 보류 FluidAudio ASR 기본값 승격 한국어 sample/meeting 전체 CER, latency, memory가 아직 없다. 측정 결과가 좋을 때만 default 후보로 올릴 수 있음

우선 적용해야 할 개선

P0 수동 benchmark를 엔진 기준으로 바꾼다
StreamingChunkBenchmarkTests, MeetingCorpusTests

현재 수동 benchmark는 STT_MODEL로 WhisperKit 모델만 바꾸기 쉽다. main에는 이미 SpeechEngineID가 있으므로 STT_ENGINE=whisper_accurate|speech_analyzer|sf_speech_on_device를 받게 바꾸면, 지금 만든 새 구조의 품질을 바로 숫자로 볼 수 있다.

성공 기준: 같은 sample/meeting에서 engine별 CER, RTF, final empty count, preview 가능 여부가 출력된다.

P1 STTService를 엔진 facade와 구현체로 나눈다
STTService.swift, SpeechEngine.swift

main의 STTService는 이미 엔진 facade 역할을 하지만 내부 구현까지 모두 들고 있다. SpeechRecognitionEngine protocol을 만들고 WhisperKit, SpeechAnalyzer, SFSpeech 구현체를 나누면 FluidAudio나 MLX sidecar를 붙일 때 변경 범위가 작아진다.

성공 기준: public API는 loadSpeechEngine, transcribe 중심으로 유지하고, 기존 SpeechEngineTests와 model cache recovery 테스트가 통과한다.

P2 VAD를 protocol로 분리한 뒤 Silero VAD를 실험한다
VADProcessor.swift, TranscriptionViewModel.swift

FluidAudio의 VadManager는 16kHz mono Float32 samples를 처리하고, 4096 sample, 256ms 단위의 speech probability를 반환한다. minto2의 MicrophoneSource는 이미 16kHz mono Float32를 만들기 때문에 입력 조건은 맞다.

다만 현재 TranscriptionViewModel은 VADProcessor()를 직접 생성한다. VoiceActivityDetector protocol과 EnergyVADProcessor adapter를 먼저 만들고, 그 다음 SileroVADProcessor를 추가해야 안전하다.

성공 기준: threshold 0.3/0.5/0.7 sweep에서 짧은 발화 recall이 개선되고 non-speech phantom이 늘지 않는다.

P3 AudioChunk와 Segment에 실제 오디오 offset을 추가한다
Meeting.swift, TranscriptionState.swift, TranscriptNormalizer.swift

현재 Segment.timestamp는 실제 오디오 시작 시각이라기보다 segment 생성 시각에 가깝다. diarization, 정확한 회의 timestamp, speaker label 매칭을 하려면 audioStartSeconds, audioEndSeconds 또는 sample range가 필요하다.

성공 기준: 기존 JSON decode를 깨지 않도록 optional/default 필드로 추가하고, normalizer가 offset을 보존한다.

P4 회의 raw audio 저장 또는 rolling buffer를 설계한다
MicrophoneSource.swift, AppDelegate.swift

Offline diarization은 전체 오디오 또는 최소한 speaker 판단에 충분한 speech region이 필요하다. 현재 앱은 transcript와 report는 저장하지만, 회의 raw audio를 영속화하는 구조는 보이지 않는다.

성공 기준: 사용자가 원하면 저장하지 않는 privacy mode도 유지하면서, diarization PoC용 raw audio 파일을 임시/로컬 캐시로 만들 수 있다.

P5 FluidAudio ASR은 hidden experimental engine으로만 추가한다
SpeechEngine.swift, STTService.swift

Qwen3-ASR, Cohere Transcribe, Nemotron 계열은 한국어 가능성이 있지만, 현재 main 기준으로는 sample/meeting 전체 CER와 memory 측정이 없다. UI에 일반 사용자 선택지로 바로 노출하면 기대 품질을 설명할 근거가 부족하다.

성공 기준: hidden/test-only engine으로 먼저 추가하고, WhisperKit/SpeechAnalyzer/SFSpeech와 같은 benchmark 표에 들어간다.

단계별 계획

단계 작업 검증 중단 기준
1 STT_ENGINE benchmark 지원 추가 RUN_STT_TESTS=1 수동 benchmark가 엔진명을 출력 Apple 엔진 availability 때문에 benchmark가 불안정하면 availability result도 같이 출력하고 skip
2 STTService 구현체 분리 swift test --filter SpeechEngineTests, model cache recovery tests public API 변경이 Settings/ViewModel 대규모 수정으로 번지면 먼저 더 작은 facade만 적용
3 VoiceActivityDetector seam 추가 기존 VADProcessorTests 통과 VAD behavior가 달라지면 adapter만 롤백하고 concrete 유지
4 FluidAudio 의존성 추가와 SileroVADProcessor PoC short utterance probe, sample/meeting VAD A/B false positive 또는 memory 부담이 커지면 production 도입 보류
5 AudioChunk/Segment offset 추가 저장 JSON 호환성, normalizer offset 보존 테스트 기존 meeting decode가 깨지면 optional migration부터 분리
6 Offline diarization PoC speaker timeline과 transcript overlap matching report speaker label이 수동 검토에서 불안정하면 UI 노출 보류
7 FluidAudio ASR hidden benchmark sample/meeting 전체 CER, RTF, peak memory WhisperKit/SpeechAnalyzer 대비 숫자가 약하면 엔진 선택지 노출 안 함

권장 결론

결론 지금 main 상태에서는 "측정 가능한 엔진 구조"부터 완성하는 것이 맞다

새 구조 덕분에 엔진 선택 UI와 기본 availability 흐름은 생겼다. 다음 작업은 새 엔진을 더 추가하는 것이 아니라, 같은 sample에서 비교 가능한 benchmark와 엔진 구현체 분리를 먼저 끝내는 것이다. 그 다음 Silero VAD, 오디오 offset, offline diarization 순서로 가면 각 개선의 효과를 분리해서 판단할 수 있다.
