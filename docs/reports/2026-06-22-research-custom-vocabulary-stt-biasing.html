01

keyword spotting vs contextual biasing — 용어 구분

이 둘은 서로 다른 파이프라인 단계를 가리킨다. 혼용하면 솔루션을 잘못 고르게 된다.

Keyword Spotting (=wake-word detection)

목적: "헤이 시리", "오케이 구글"처럼 특정 단어가 발화됐는지 탐지하는 별도 task. 전사(transcription)와 무관한 파이프라인이다. 경량 binary classifier로 구현되며, 결과는 "발화됨 / 발화 안 됨"이다. Picovoice Porcupine, Apple SpeechDetector 등이 이 역할을 한다.

Contextual Biasing (= vocabulary biasing / custom vocabulary)

목적: 전사 모델이 특정 단어를 더 잘 인식하도록 디코딩 중 확률 분포를 조정하는 기법. "나무호", "쿠팡", "FluidAudio" 같은 도메인 고유명사의 전사 오류를 줄인다. minto2가 필요한 건 이것이다.

결론
"keyword spotting" 기술을 용어 전사 정확도 개선에 쓸 수는 없다. 우리 목표는 contextual biasing이다. 이하 조사는 이 관점으로 진행한다.
02

WhisperKit의 vocabulary / biasing API

DecodingOptions — biasing 관련 필드 전수
// Sources/WhisperKit/Core/Configurations.swift (argmaxinc/argmax-oss-swift)
public struct DecodingOptions: Codable, Sendable {

  // ── 바이어싱 관련 ──────────────────────────────────────
  var promptTokens:  [Int]?   // 이전 컨텍스트 프롬프트. prefill 토큰 앞에 삽입
  var prefixTokens:  [Int]?   // 현재 발화 초기 프리픽스. prefill 토큰 뒤에 추가
  var suppressTokens: [Int]   // 디코딩 중 억제할 토큰 ID 목록
  var suppressBlank:  Bool    // 공백 토큰 억제 여부

  // ── 품질 게이팅 (간접 영향) ────────────────────────────
  var logProbThreshold:          Float?  // 평균 log-prob 하한 (기본 -1.0)
  var temperatureFallbackCount:  Int     // fallback 온도 증가 횟수 (기본 5)

  // ── 미지원 (로드맵에만 있음) ────────────────────────────
  // logit bias / word boosting → Roadmap: "multi-token forward pass" 필요
}
공백 확인
logit bias(단어별 확률 직접 조정)는 공개 API에 없다. WhisperKit Roadmap에 "multi-token TextDecoder forward pass"가 advanced feature로 기재돼 있으며 아직 구현 전이다.
promptTokens vs prefixTokens — 차이와 용도
prompt이전 30초 컨텍스트 역할. Whisper가 스타일·형식·용어를 유지하도록 컨디셔닝. 토큰 시퀀스 앞에 삽입됨. 용어 바이어싱에 권장.
└ 제약: 마지막 224 토큰만 유효. KV 캐시 무효화 → 첫 토큰 지연 소폭 증가.
prefix현재 발화의 부분 전사를 이어받을 때 사용. 실시간 스트리밍 오버랩 구현 등에 활용. 도메인 용어 강제에는 부적합.
└ 용어 바이어싱 목적으로는 설계된 필드가 아님.
확인됨
용어 바이어싱에는 promptTokens가 적합한 필드다. 사용 방법: 용어를 tokenizer로 인코딩 후 특수 토큰 필터링, DecodingOptions(promptTokens:)에 전달. 실측에서 "나무어→나무호" 교정 확인(비-turbo large-v3). 
// 용어 바이어싱 패턴 (검증됨 — 비-turbo large-v3)
let terms = ["FluidAudio", "나무호", "쿠팡"]
let promptText = terms.joined(separator: ", ")
let allTokens = tokenizer.encode(text: promptText)
let promptTokens = allTokens.filter { $0 < tokenizer.specialTokens.specialTokenBegin }

var options = DecodingOptions()
options.promptTokens = promptTokens
// options.usePrefillPrompt = true  ← 기본값 유지

large-v3-turbo에서 promptTokens 빈 출력 이슈

우리가 실측 확인한 현상 — turbo에서 promptTokens 사용 시 전사 결과가 empty — 이 GitHub에서도 동일하게 보고돼 있다. 실측 확인

Issue #372argmaxinc/WhisperKit(→ argmaxinc/argmax-oss-swift) 에 대한 미해결 버그 보고. large-v3 계열 모델에서 promptTokens 사용 시 빈 출력 반환. PR #438이 연결돼 있으나 수정 내용 미확인.
└ 상태: Open (2025년 10월 보고, 2026년 6월 기준 미해결 확인)
핵심 제약
turbo에서 promptTokens가 깨지는 근본 원인은 공식 문서에서 밝혀지지 않았다. 가설: turbo는 decoder 레이어가 줄어 prompt conditioning pathway가 다르게 작동할 수 있음. 추정
prefixTokens는 turbo에서 동작하는가?

공식 문서 및 Issue #372 어디에도 prefixTokens가 turbo에서 정상 동작한다는 증거가 없다. 미확인

또한 prefixTokens는 용어 바이어싱이 아닌 스트리밍 연속성 목적이므로, 기능적으로도 우리 용도와 맞지 않는다.

현실적 선택
turbo + 용어 바이어싱 조합은 현재 지원되지 않는다. 비-turbo large-v3 + promptTokens가 현재 유일하게 동작이 확인된 WhisperKit 내 경로다. 속도 패널티가 존재하므로 교정 레이어 병행을 고려해야 한다.

Apple Speech 프레임워크의 custom vocabulary

SFSpeechRecognizer + contextualStrings (구형, iOS 10+ / macOS 10.15+)

SFSpeechRecognitionRequest.contextualStrings는 시스템 어휘에 없는 단어·구문 목록을 전달해 인식 확률을 높이는 lightweight biasing API다. 공식 문서

파라미터[String]. 각 항목은 짧은 구문(단어~구 수준).
개수 한계공식 문서에 명시적 상한 없음. 실무 커뮤니티 기준 약 100개 이하. 초과 시 무시되거나 성능 저하. 추정
온디바이스requiresOnDeviceRecognition = true 필수. 미설정 시 서버 경로 사용 → contextualStrings 미적용 위험.
macOSmacOS 10.15+ 지원. macOS 14에서 사용 가능. 확인
한국어ko-KR 로케일로 SFSpeechRecognizer.supportedLocales()에 포함돼 있으나, contextualStrings의 한국어 바이어싱 실효성은 공식 문서에서 확인되지 않음. 미확인
정확도Apple 내부 모델 기반. Whisper large-v3 대비 WER 열세(Apple 14.0% vs WhisperKit small.en 12.8%). 한국어 회의 도메인 비교치 없음. 추정
중요 제약
공식 문서에서 명시: "contextualStrings는 완전한 custom vocabulary workflow의 대체제가 아니다." 희귀 고유명사 강제에 제한적이며, Whisper의 prompt 방식보다 통제력이 낮다.
SFSpeechLanguageModel.Configuration — 더 강력한 custom vocabulary (iOS 17+ / macOS 14+)

WWDC23에서 발표된 신기능. 앱이 도메인별 학습 데이터(PhraseCount + 발음 기호)를 준비하면 prepareCustomLanguageModel()이 온디바이스에서 언어 모델을 파인튜닝한다. WWDC23 공식

동작구문 샘플 + 발음 기호(X-SAMPA) → 파일로 내보내기 → 앱 에셋 배포 → prepareCustomLanguageModel() 호출(백그라운드 필수, 지연 큼) → 요청에 연결.
macOSiOS 17+에서 도입. macOS 14 지원 여부는 공식 문서에서 명시되지 않음. 미확인
한국어로케일별로 지원 X-SAMPA 부분집합이 다르다고 명시. ko-KR 발음 기호 지원 여부 미확인. 미확인
제약온디바이스 강제(requiresOnDeviceRecognition = true). 단일 로케일 바인딩. 모델 준비 지연이 크다. 회의 도중 실시간 적용 불가.
macOS / 한국어 미확인
회의록 앱(macOS 14, 한국어) 관점에서 가장 중요한 두 조건 모두 공식 확인이 없다. 사용 전 Xcode에서 직접 API 가용성 확인이 필요하다.

2. custom vocabulary 미지원 → 도메인 용어 바이어싱 목적 자체를 달성할 수 없다.

권고 — 우리 스택에 적합한 경로

1순위
promptTokens (비-turbo large-v3) + 교정 레이어
현재 유일하게 동작이 확인된 WhisperKit 내 biasing 경로. 비-turbo의 속도 패널티를 교정 레이어가 보완한다. 이미 두 컴포넌트가 존재하므로 통합 비용이 낮다. 용어집에서 상위 N개 용어를 추출해 promptTokens로 전달하는 파이프라인을 구성한다.
즉시 적용 가능 macOS 14+ 한국어 속도 패널티 있음
2순위
교정 레이어 강화 (단독)
STT 모델을 비-turbo로 전환하기 어려운 경우의 대안. 용어집 기반 LLM 교정이 이미 구현돼 있으므로 promptTokens 없이도 후처리만으로 도메인 용어 오류를 상당 부분 잡을 수 있다. 속도 이점을 유지한 채 정확도를 높이는 현실적 트레이드오프.
즉시 적용 가능 turbo 유지 biasing 아님 (교정)
3순위
SFSpeechLanguageModel.Configuration 검증 (중장기)
macOS 14 + ko-KR 지원 여부를 Xcode에서 직접 확인한 뒤, 유효하면 회의 시작 전 백그라운드 준비 파이프라인 구성을 검토한다. 준비 지연이 크므로 실시간 전사 중 동적 적용은 불가하다. WhisperKit을 대체하지 않고 병행 경로로만 고려한다.
macOS 14 미확인 한국어 미확인 준비 지연 큼
보류
SpeechAnalyzer, turbo+promptTokens, prefixTokens를 용어 바이어싱에 쓰는 것은 각각 macOS 버전 불일치·알려진 버그·용도 불일치로 현재 권고하지 않는다.

구현 경로 — 확인해야 할 것

즉시비-turbo large-v3에서 promptTokens 파이프라인을 현재 turbo 경로와 나란히 두고, 모델 선택과 promptTokens 여부를 설정으로 분리해 A/B 테스트 가능하게 구조화한다.
단기용어집 상위 N개 추출 → 용어 빈도·희귀도 기반 정렬 → prompt 224 토큰 예산 내 최적 구성. 현재 용어집 추출 로직이 있으면 연결.
검증WhisperKit Issue #438 PR을 추적해 turbo + promptTokens 수정이 병합되면 재테스트. turbo 복귀 판단 기준을 미리 정의해 둔다(속도 vs 정확도 임계치).
중기SFSpeechLanguageModel.Configuration의 macOS 14 + ko-KR 가용성을 Xcode에서 API 존재 여부로 확인. 유효시 별도 ADR 작성 후 검토.
