# STT 벤치마크 용어집 (비전문가용)

STT 엔진 비교/벤치마크에서 쓰는 용어를 **받아쓰기 시험 비유**로 풀어 정리한다.
이 분야가 처음이어도 따라올 수 있게, 모든 용어를 정의와 예시로 적는다.

> 사용법은 [`tools/stt-benchmark/README.md`](../tools/stt-benchmark/README.md), 설계 근거는 [ADR 0004](adr/0004-stt-benchmark-reliability-methodology.md).

---

## 큰 그림 — "받아쓰기 시험" 비유

여러 STT 프로그램(엔진) 중 **어느 것을 Minto에 쓸지** 고르는 일은, 학생 여러 명에게 **같은 받아쓰기 시험**을 보게 하고 누가 제일 잘하는지 가리는 것과 같다.

| 비유 | 실제 |
|---|---|
| 받아쓰기 보는 학생 | STT **엔진** (whisper, speech_analyzer 등) |
| 들려주는 음성 | 회의 녹음 파일 |
| 학생이 적은 답안 | 엔진이 만든 **전사 결과** |
| 채점 기준(정답지) | **레퍼런스** (회의 자막 SMI) |
| 틀린 글자 비율 | **CER** (점수) |

핵심 어려움 두 가지가 이 용어들을 만들었다:
1. **같은 학생도 시험마다 점수가 흔들린다** → 반복 측정·신뢰구간이 필요.
2. **정답지가 완벽하지 않다** → 절대 점수는 못 믿고, 상대 비교만 유효.

---

## 1. 기본 용어

**STT** (Speech-to-Text)
: 음성을 글자로 바꾸는 기술. "음성 인식"과 같은 말.

**엔진 (engine)**
: STT를 실제로 수행하는 프로그램/모델. 후보들: `whisper_accurate`(WhisperKit), `speech_analyzer`/`sf_speech_on_device`(애플), `nemotron`, `sherpa`. = 받아쓰기 보는 **학생**.

**전사 (transcription)**
: 엔진이 음성을 듣고 만든 **글자 결과물**. = 학생이 적은 **답안**.

**CER** (Character Error Rate, 글자 오류율)
: 정답지 대비 **틀린 글자의 비율**. 낮을수록 잘함. 예: `CER 0.30` = 글자의 30%가 틀림. = 받아쓰기 **점수**(틀린 비율).

**레퍼런스 (reference) / 정답지**
: 채점 기준이 되는 "정답 글". 우리는 회의 영상의 **자막 파일(SMI)**을 정답지로 쓴다.

**SMI**
: 방송용 자막 파일 형식. 우리 정답지가 이 형식이다.

**verbatim / non-verbatim**
: **verbatim** = 말한 것을 **한 글자도 안 빼고 그대로** 적은 것(축자 전사). **non-verbatim** = 그게 **아닌 것** — 말을 요약·정리·생략해 적은 것.
: ⚠️ 우리 정답지(SMI 자막)는 **non-verbatim**이다. 방송 자막이라 말을 다듬어 적었다. 그래서 엔진이 말을 완벽히 받아써도 정답지(요약본)와 안 맞아 "틀림" 처리된다 → **잘해도 CER이 40~50%대 밑으로 잘 안 내려간다**(이걸 "floor(바닥)"라 부른다).

## 2. 신뢰성 용어 (점수가 흔들리는 문제)

**ANE / 비결정성 (non-determinism)**
: ANE = Apple Neural Engine(애플 신경망 칩). 엔진이 이 칩 위에서 도는데, 미세하게 **비결정적**이라 **같은 음성·같은 엔진도 돌릴 때마다 결과가 조금씩 다르다**(관측상 ±8%P 정도). = 같은 학생이 같은 시험을 봐도 매번 점수가 약간 다른 것.

**N (반복 횟수)**
: 같은 엔진에게 **같은 측정을 몇 번 반복**시키는가. 한 번만 재면 그 점수가 우연인지 실력인지 모르므로 N번 반복해 평균을 본다. 현재 잠정 N=5.

**CI** (Confidence Interval, 신뢰구간)
: 점수가 매번 흔들리니까, "진짜 실력은 **이 범위 안**"이라고 통계로 잡은 점수 범위. 예: `0.47 ± 0.02` → 실력은 0.45~0.49 사이.
: ⚠️ 배포할 때 쓰는 **CI(Continuous Integration)와 전혀 다른 말**이다. 여기선 항상 "신뢰구간"을 뜻한다.

**무승부 (tie)**
: 두 엔진의 신뢰구간(점수 범위)이 **겹치면**, 점수 차이가 흔들림 안에 묻힌 것이라 "누가 낫다" 단정 못 한다 → **무승부**로 본다. 예: A가 0.45~0.49, B가 0.47~0.51이면 겹치므로 무승부. CER 평균이 1등이어도 2등과 범위가 겹치면 확정 1등이 아니다.

## 3. 판정 용어 (그래서 채택하나?)

**improvement_signal (개선 신호)**
: 후보 엔진이 기준 엔진보다 어떤지 신뢰구간으로 낸 판정.
: `significant_improvement`(확실히 개선 — 후보 범위가 기준 범위보다 통째로 아래) / `significant_regression`(확실히 악화) / `tie`(무승부) / `unknown_ci`(반복측정 안 해 범위를 모름).

**sanity 상한 (sanity ceiling)**
: "이 점수보다 나쁘면 측정이 깨졌거나 쓸 수 없는 품질"로 보는 **상한선**(현재 CER 0.70). 후보가 다른 후보보다 낫더라도 이 선을 넘으면 채택 거부. = "둘 다 낙제점이면 그중 1등이어도 안 뽑는다."

**regression gate (악화 감지 관문)**
: 후보가 기준보다 **나빠지지 않았는지** 검사하는 단계. 여기서 개선/악화 신호를 만든다.

**decision gate (채택 판정 관문)**
: 최종적으로 **채택할지** 정하는 단계. 결과(`decision_state`):
: `default_allowed`(제품 기본으로 **교체 권장**) / `experimental_flag_only`(실험 기능으로만, 기본은 유지) / `rejected`(채택·실험 모두 불가).

**lane matrix (순위표)**
: 모든 엔진을 점수순으로 줄 세운 표. 무승부면 같은 등급으로 묶는다.

## 4. 품질 함정 용어

**phantom (환각, hallucination)**
: 엔진이 **안 들린 구간(침묵 등)에 없는 말을 지어내는** 것. AI 음성인식의 고질병.
: ⚠️ 묘한 함정: 침묵을 비워두면 채점에 불리한데, **그럴듯하게 지어내면 오히려 점수가 좋아 보일 수** 있다 → 채점을 속인다. (현재 우리 게이트는 이걸 아직 못 잡는다 — 알려진 구멍.)

**ground-truth (정답 사실 데이터)**
: "진짜로 이게 사실"이라고 사람이 확정해둔 데이터. 환각을 잡으려면 "여기는 원래 아무 말도 없는 구간"이라는 ground-truth가 필요한데, 아직 안 만들었다.

## 5. 측정 경로 — product-path (⚠️ 현재 조정 필요)

**하니스 경로 (test harness)**
: 테스트용으로 깔끔하게 측정하는 일반 경로. 비교/랭킹의 기본값.

**product-path (실제 제품 경로)**
: 연습장이 아니라 **실제 Minto 앱이 음성을 처리하는 그대로**의 경로로 측정하는 모드. 엔진을 진짜 **채택**하려면 "실제 앱에서도 잘 되나"를 봐야 하므로, `default_allowed`(교체 권장) 판정은 이 경로를 요구한다.

**⚠️ 현재 문제 (조사 완료 — 원인: VAD):**
: 같은 8분 회의를 측정했을 때 —
: - 하니스 경로: **1813글자** 전사, CER **0.49** (정상)
: - product-path 경로: **413글자**만 전사, CER **0.89**
:
: **원인 추적 결과(2026-06-23):**
: - 하니스 경로는 클립을 `service.transcribe()`로 **VAD 없이 통째 전사**한다.
: - product-path는 실제 앱처럼 오디오를 **VAD(음성 구간 검출)에 먼저 통과**시키고 살아남은 구간만 STT에 넘긴다. 이 VAD가 대부분을 잘라내 413자만 남는다.
: - 기본 VAD는 **에너지 기반(VADProcessor)** — 음량 적응형 임계값으로 조용한 발화를 침묵으로 보고, **0.5초 미만 발화는 버린다**(`minSpeechSamples` 게이트).
: - 후보에서 제외된 것: evict 캡(5000이라 8분 회의는 미발동), 오디오 피드 드롭(emit→VAD 동기 호출, drain이 STT 완료까지 대기), 측정 위치 오류.
:
: **실제 앱 영향(미확정 — 검증 필요):** 에너지 VAD는 production 기본값이라 이 경로는 실제다. 단 벤치마크는 **녹음된 깨끗한 WAV를 짧은 창으로 잘라 먹이는데**, 에너지 VAD가 발화로 꽉 찬 클립에서 노이즈 floor를 잘못 보정해 **실제 마이크 녹음보다 더 많이 자를** 수 있다. 즉 413/1813은 라이브 사용보다 과장됐을 가능성. **다음 결정적 실험**: ① `MINTO_VAD_ENGINE=silero`로 재실행 → 글자 수가 회복되면 에너지 VAD가 범인. ② 창별 VAD 통과 길이를 계측. ③ 실제 라이브 녹음과 비교.
: **이건 측정 프레임워크가 아니라 앱(Swift)의 VAD 처리 경로** 문제다.

## 한눈에 — 신뢰도 요약

- **노이즈에 안 속고 A vs B 우열 가리기** → 신뢰 가능(반복측정 + 신뢰구간으로 단일측정 착시 차단).
- **절대 점수를 "정확도"로 읽기** → 불가(정답지가 non-verbatim).
- **최종 채택 판정** → 아직 부족(N 미보정, 실측 부족, phantom 미반영, product-path 깨짐).
